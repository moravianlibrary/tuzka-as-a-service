# taas 0.6.x — implementation status & review guideline

> Status + review-handoff doc. Written 2026-06-30. Companion to the design doc
> [2026-06-18-job-analytics-design.md](2026-06-18-job-analytics-design.md) (the spec) and
> [2026-06-29-analytics-query-scaling.md](2026-06-29-analytics-query-scaling.md) (a known
> deferral). Read those first for *intent*; this doc records *what is built* and *how to
> check it*.

## TL;DR

The 0.6.x release (analytics fact table + per-job metrics + domain routing + queue
priority + per-box exporter tunneling + dashboard UX) is **implemented end-to-end** and
verified (unit + integration tests against a real Postgres, plus a live k3d run of the
dashboard/API). One item is **deferred on purpose** (the daily rollup), and **release
mechanics** (version bump + commit) are not done yet. All changes are in the working tree,
uncommitted, on `main`.

---

## What "done" means here

A feature is DONE only if: code path exists, it has test coverage (unit and/or the
`TEST_DATABASE_URL` integration suite), and — for anything user-facing — it was driven in
the running app, not just unit-tested.

## Deliberately NOT done (do not flag as missing)

1. **`job_analytics_daily` rollup** — heavy admin reporting queries are slow at ~10M rows.
   Documented + accepted in `2026-06-29-analytics-query-scaling.md`. Interim mitigation:
   keep CSV/stats exports range-bounded.
2. **Release mechanics** — Chart `appVersion`/`pyproject` version bump, changelog, and the
   commit itself are NOT done. The working tree is uncommitted on `main`.

## Review history (all resolved)

A high-effort review (7 finder angles + verify) produced 9 findings. Resolution:

- **#1** catalog link not scheme-gated → fixed: `isHttpUrl()` gates to http(s) in `dashboard.js`.
- **#5** breakdown pager over-reported "Next" (LIMIT 50, no probe) → fixed: endpoint now
  `LIMIT 51` + returns `has_next`; JS uses it. Covered by `test_breakdown_pagination_has_next`.
- **#6** `to_date` dropped the final day (date-input midnight) → fixed: dashboard sends
  end-of-day for breakdown + CSV.
- **#2** Service-name length unchecked → fixed: `fail` when `>63` chars.
- **#4** `gpu-exporter` host only in the GPU overlay → documented in box README (⚠️ note).
- **#7** breakdown error cell showed `[object Object]` on 422 → fixed: `errText()` formatter + escaping.
- **#8** `analytics_raw` / `analytics_raw_csv` duplicated WHERE/FROM/params → fixed:
  `_ANALYTICS_FROM` + `_analytics_filters()`.
- **#9** dead `app/services/stats.py` + `app/models/job_daily_stats.py` → deleted; README prose updated.
- **#3** Feature-6 iteration/naming/validation duplicated across 4 Helm templates → fixed:
  shared `_helpers.tpl` defines (`taas.tunnelEngineService`, `taas.tunnelExporterService`,
  `taas.validateTunnelBoxes`).

**Bug found during verification (not in the original review):** the exporter ServiceMonitor
selected on `app.kubernetes.io/name`+`instance`, but the exporter Service's labels used only
`taas.labels`, which on this chart omits those keys → the Operator would scrape nothing.
Fixed by adding `taas.selectorLabels` to the tunnel Service labels. See the
`helm-labels-vs-selectorlabels` project memory.

## Post-review fixes (2026-06-30)

A second review pass produced fixes now applied to the working tree:

- **Analytics write isolation (correctness).** `write_analytics_row` now runs inside a
  SAVEPOINT (`db.begin_nested`), so a failed analytics insert rolls back only that row
  and can no longer poison the harvest transaction (which had stored the OCR result and
  staged the `jobs` update) and wrongly mark a successful job `failed`.
- **Dashboard XSS.** The analytics breakdown table now `escapeHtml`s `username` /
  `engine_version` / `engine_device` / `domain` (matching the backends tab).
- **Submit-worker domain routing.** Served-domain set is loaded once per backend per tick
  (no per-job DB round-trip); domain sync is a tracked, per-backend-gated background task
  (was an untracked `ensure_future` that could be GC'd / race concurrent rebuilds).
- **Timezone.** Analytics/jobs date filters normalize tz-aware query params to naive UTC
  (`_naive_utc`), so range filters are correct regardless of the PG session timezone.
- **CSV.** Both CSV exports defuse spreadsheet formula injection (`_csv_cell`); the always-0
  `requeues_total` column was removed from `stats.csv` (requeues aren't in analytics).
- **Cleanup.** Dropped the unused `JobAnalytics`/`EngineVersion` ORM models; moved
  function-local `datetime` imports to module scope; added module headers to the new models.

### Known divergences from the design doc (accepted)

- Lookup PKs are `INTEGER`, not `SMALLINT` (no overflow risk; storage estimate slightly
  higher than the design's "SMALLINT FKs" note).
- `job_analytics.fmt` is plain `TEXT`, not a `job_fmt_t` enum (the enum was never created).
- Analytics queries are hand-written SQL (`text()`) rather than ORM — deliberate, for the
  percentile/window aggregates; all user values are bound params (the only interpolation,
  `_alto_range`, uses hardcoded ints).

## How to verify (commands)

**Pure unit tests (no infra):**
```
python -m venv /tmp/taas-venv && /tmp/taas-venv/bin/pip install -e '.[dev]'
/tmp/taas-venv/bin/python -m pytest tests/test_analytics_parse.py -q
/tmp/taas-venv/bin/ruff check app/ tests/
```

**Integration tests (need a migrated Postgres):** make a throwaway DB on the local k3d
cluster PG, then point `TEST_DATABASE_URL` at it — full recipe in the
`run-db-tests-against-k3d` memory. Expected: `pytest -q` → all pass, only the 6 Redis
rate-limit tests skip when Redis isn't forwarded.

**Helm:**
```
helm lint deploy/helm/taas
helm template taas deploy/helm/taas                       # default (no boxes) renders
helm template taas deploy/helm/taas -f <multi-box values> # engines+exporters+ServiceMonitor
```
Render a values file with two boxes, multiple engines, and exporters; confirm: one Service
per engine (`*-tunnel-engine-<box>-<engine>`) and per exporter (`*-tunnel-box-<box>-<exporter>`),
`allowPorts` lists every engine+exporter port, register-job URLs match the Service names,
and each ServiceMonitor `spec.selector.matchLabels` is a subset of its Service's labels.
Negative checks: a duplicate `remotePort` and a >63-char Service name must both `fail`.

**Live app (UI changes):** rebuild the api image, import to k3d, restart `taas-api`
(see the `local-kube-deploy-iterate` memory), then drive the dashboard / `/dashboard/*`
endpoints. The static surface (`dashboard.js`, `index.html`) ships in the **api** image;
workers + `analytics.py` ship in the **worker** image.

## Review checklist for the next pass

1. **Spec conformance** — diff the implementation map above against
   `2026-06-18-job-analytics-design.md`; flag any feature whose behavior diverges from the
   design (not just "missing", but "does something different").
2. **Analytics correctness** — timing columns (`system_queue_s`/`engine_queue_s`/
   `ocr_running_s`/`time_in_system_s`) vs the lifecycle timestamps; NULL handling for failed
   jobs that never reached a stage; ALTO category → BETWEEN mapping; enum casts.
3. **Hot path** — `submit_job` and the poller run per job; confirm no avoidable extra DB
   round-trips or row multiplication were introduced.
4. **Helm** — render with adversarial values (empty lists, one box no exporters, long names,
   duplicate ports across boxes); confirm validation fires and label-selectors match.
5. **Security** — admin-set free text reaching the dashboard (`external_url_template`),
   SQL built by string concatenation in the analytics endpoints (`_analytics_filters` /
   `_alto_range` — are all user values bound params, not interpolated?).
6. **Deferrals** — confirm the items in "deliberately NOT done" are still acceptable; don't
   re-flag them as bugs. (`mean_conf` is NOT a deferral — it's implemented in the vendored
   engine; verify the full chain pipeline→status→poller→`job_analytics.mean_conf` if in doubt.)
